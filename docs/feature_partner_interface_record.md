# 신규 기능 구현 기록: 제휴사 및 인터페이스 정책 관리 분리 및 CRUD 구현

## 구현 일자: 2026-04-26

### 📌 기획 배경 및 목적

초기 기획에서 제휴사 정보에 통신 프로토콜과 API URL이 혼재되어 있었습니다.
비즈니스 성격의 '제휴사 연락처 정보'와 시스템 성격의 '인터페이스 연동 정책(프로토콜, URL, 타임아웃)'을 명확히 분리하여, 단일 책임 원칙을 준수하고 시스템 확장성을 높이기 위해 도메인을 분리 및 고도화했습니다.

### ⚙️ 핵심 구현 사항

1. **도메인 분리 및 백엔드 CRUD API 구축**
   - **제휴사(Partner)**: 담당자명, 연락처 등 비즈니스 정보 관리 용도로 재편.
   - **인터페이스 정책(Interface Config)**: 대상 시스템, 프로토콜, API URL, 타임아웃, 최대 재처리 횟수 관리 용도로 신설.
   - Spring Data JPA를 활용해 각각 독립된 Entity, Repository, Service, Controller 계층을 완벽히 분리해 구축했습니다.

2. **UI/UX 뷰(View) 및 모달(Modal) 고도화**
   - 기존 인라인 입력 폼 대신 팝업 형태의 **신규 등록 / 수정 모달**을 일관되게 적용하여 사용자 입력 공간을 확보했습니다.
   - 등록된 제휴사 정보를 인터페이스 정책 등록 폼의 '연동 제휴사' Dropdown(`select`)과 연동하여 프론트엔드 단에서 참조 무결성을 강화했습니다.

3. **장애 시뮬레이터 기능 확장**
   - 기존 에러(Error)와 지연(Delay) 외에 **정상(Success, 200/201)** 상태 코드를 시뮬레이션할 수 있도록 발생 유형 옵션을 추가했습니다.
   - 이를 통해 에러 관제뿐만 아니라, 통합 플랫폼의 정상 트랜잭션 흐름도 완벽히 테스트할 수 있게 되었습니다.
