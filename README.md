## ✨ 핵심 기능 (Key Features)

### 1. 📊 실시간 관제 대시보드

- **자동 갱신 & 무한 스크롤**: 5초마다 백그라운드에서 최신 트랜잭션 데이터를 동기화(Polling)하며, `IntersectionObserver`를 활용한 무한 스크롤로 브라우저 성능 저하 없이 대량의 데이터를 시각화합니다.
- **검색 및 다중 필터링**: Trace ID나 기관명으로 즉시 검색하고, 상태별(정상, 장애, 지연) 필터링 및 프로그레스 바 차트를 실시간으로 제공합니다.

### 2. 🚨 장애/미처리 전용 대기열 (DLQ)

- **원인별 격리 & 필터링**: 500(서버 에러), 504(타임아웃) 등 응답 코드별로 실패/지연 건을 모아 집중 분석합니다.
- **일괄 재처리 (Bulk Retry)**: 장애 건을 다중 선택하여 비동기(`Promise.all`)로 일괄 재전송을 요청하거나, 영구 실패 건을 '처리 포기'하여 대기열에서 소거합니다.

### 3. ⚙️ 제휴사 및 인터페이스 정책 (Config) 관리

- **도메인 분리 & 컴포넌트화**: 담당자 연락처를 관리하는 '제휴사'와 라우팅/타임아웃을 관리하는 '인터페이스 정책'을 분리하고, 순수 Vanilla JS 컴포넌트로 모듈화하여 관리합니다.
- **참조 무결성 UX**: 제휴사가 추가/수정되면 인터페이스 정책 등록 폼의 드롭다운 목록이 실시간으로 동기화됩니다.

### 4. 📝 운영 작업 이력 (Audit Log)

- **컴플라이언스 추적**: 수동 재처리, 처리 포기, 정책 등록/수정/삭제 등 시스템 상태를 변화시키는 모든 운영자의 액션을 백엔드 DB에 자동 기록하여 투명하게 조회합니다.

### 5. 🧪 능동형 장애 시뮬레이터

- **임의 시뮬레이션 자동 발생**: 등록된 제휴사/정책을 기반으로 정상(60%), 장애(20%), 지연(20%) 상태의 트랜잭션을 2초 간격으로 연속 5회 자동 생성하여 모니터링 반응을 테스트할 수 있습니다.
- **수동 에러 주입**: HTTP 상태 코드 및 JSON 페이로드 전문을 직접 작성하여 세밀한 엣지 케이스를 재현합니다.

### 6. 🛡️ 엣지 케이스 방어 및 편의 기능

- **위험 알림 (Alerting)**: 미처리 장애 건수가 3건 이상 누적되면 화면에 즉각적으로 경고 팝업을 띄웁니다.
- **데이터 마스킹 & CSV 다운로드**: 모달창에서 민감 정보를 `**`로 마스킹 처리하며, 조회된 데이터 목록을 즉시 엑셀(CSV) 형태의 파일로 추출할 수 있습니다.
- **동적 URL 라우팅**: 로컬 및 운영 배포 환경을 프론트엔드가 자동 감지하여 백엔드 API 주소를 유동적으로 변경합니다.

# 🏃 Noa Insure: 핵심 사용자 시나리오 (User Scenarios)

## 👨‍💻 페르소나 (Persona)
* **이름:** 김관제 (IT 인프라 운영팀 매니저)
* **역할:** 사내 코어 시스템과 외부 기관(제휴사, 금감원 등) 간의 데이터 연동이 끊김 없이 24시간 정상 동작하는지 감시하고, 장애 발생 시 최단 시간 내에 복구하는 역할.

---

## 🟢 시나리오 1: 평시 모니터링 및 자동 갱신 (Daily Monitoring)
1. **[접속]** 출근 후 김 매니저는 'Noa Insure' 대시보드를 띄워둡니다.
2. **[실시간 감시]** 별도로 새로고침을 누르지 않아도, 백그라운드 자동 갱신(Polling) 기능이 5초마다 최신 트랜잭션 데이터를 불러옵니다.
3. **[시각적 인지]** 화면 중앙의 '트랜잭션 상태 비율 현황(Progress Bar)'이 녹색(정상) 99%를 유지하고 있는 것을 보며 시스템이 안정적임을 직관적으로 확인합니다.

## 🚨 시나리오 2: 장애 조기 감지 및 신속 대응 (Incident Response & DLQ)
1. **[위험 감지]** 점심시간 직후, 업비트 시세 연동 API 측 서버 문제로 `504 Gateway Timeout` 에러가 연속 발생합니다.
2. **[알림 발생]** 시스템이 미처리 장애 건수 증가를 감지하고 화면 우측 하단에 **"⚠️ [장애 감지] 새로운 통신 장애가 발생했습니다!"** 라는 붉은색 경고 토스트를 띄웁니다.
3. **[원인 분석]** 김 매니저는 즉시 **'장애/미처리 대기열(DLQ)'** 탭으로 이동하여, 필터를 `504 Timeout`으로 설정합니다. 타임아웃으로 실패한 건들만 리스트업 됩니다.
4. **[상세 확인]** Trace ID를 클릭하여 팝업(Modal)을 띄운 뒤, 마스킹(Masking) 처리되어 개인정보가 가려진 안전한 JSON 페이로드 전문과 에러 코드를 확인하여 외부 통신망 장애임을 확정합니다.

## 🔄 시나리오 3: 일괄 복구 및 처리 포기 (Bulk Retry)
1. **[복구 시도]** 10분 뒤, 외부 제휴사(업비트)의 서버가 복구되었다는 연락을 받습니다.
2. **[일괄 재처리]** 김 매니저는 대기열(DLQ) 화면에서 아까 실패했던 504 에러 건들의 체크박스를 '전체 선택'한 후, **[선택 일괄 재처리]** 버튼을 클릭합니다.
3. **[결과 확인]** 병렬 비동기(`Promise.all`)로 서버에 재전송이 요청되며, 모두 'SUCCESS'로 처리되어 대기열 큐에서 깔끔하게 사라집니다. (의미 없는 과거 데이터는 '선택 처리 포기'를 눌러 큐를 비웁니다.)

## 🏢 시나리오 4: 신규 제휴사 연동 및 정책 설정 (Config Management)
1. **[제휴사 등록]** 회사에 새로운 간편결제 파트너인 '토스페이먼츠'가 추가되었습니다.
2. **[비즈니스 정보 입력]** 김 매니저는 **'제휴사/고객사 현황'** 탭에서 `[+ 신규 등록]`을 눌러 토스페이먼츠의 담당자 이름과 연락처를 등록합니다.
3. **[기술 정책 설정]** 이어서 **'인터페이스 관리'** 탭으로 이동해 신규 정책을 등록합니다. 방금 등록한 '토스페이먼츠'를 드롭다운에서 선택하고, 통신 프로토콜을 `REST API`로, 타임아웃을 `4000ms`, 최대 재처리 횟수를 `3회`로 설정하여 저장합니다.

## 📝 시나리오 5: 보고서 추출 및 컴플라이언스 대응 (Audit & Export)
1. **[이력 추적]** 주간 회의 시간, 보안팀에서 "어제 오후 2시경 대량 재처리를 수행한 내역과 원인"을 요구합니다.
2. **[작업 이력 확인]** 김 매니저는 **'운영 작업 이력(Audit Log)'** 탭에 들어가 어제 날짜에 기록된 `RETRY` 액션과 자신의 식별자, 상세 사유가 DB에 위변조 없이 자동 기록되어 있는 것을 확인하고 소명합니다.
3. **[데이터 추출]** 회의 자료 첨부를 위해 대시보드에서 어제 날짜로 검색(필터링)한 뒤, 우측 상단의 **[엑셀(CSV) 다운로드]** 버튼을 클릭하여 즉각적으로 보고서를 생성합니다.



## 🛠️ 기술 스택 & 아키텍처 (Tech Stack & Architecture)

### Frontend

- **Framework:** Vanilla JS (의존성 패키지 없이 순수 웹 표준 기술)
- **Architecture:** Component 기반 설계 (`view-partner.js`, `view-policy.js`) 및 SPA 라우팅
- **Styling:** CSS3 (Flexbox/Grid 기반 완벽한 반응형 대응)

### Backend

- **Framework:** Java 21, Spring Boot 3.x
- **Database:** H2 In-memory DB / Spring Data JPA (Hibernate)
- **Architecture:** REST API, Layered Architecture (Controller - Service - Repository)
- **Security:** `.env` 템플릿화를 통한 DB 접속 정보 등 환경변수 분리

## 실행 방법 (Getting Started)

본 프로젝트는 프론트엔드와 백엔드가 철저히 분리된 모노레포 구조를 가집니다.

### 💻 Frontend 실행

별도의 빌드 과정 없이 정적 파일만으로 구동됩니다.

1. `front/` 폴더로 이동합니다.
2. `index.html` 파일을 브라우저로 드래그하여 열거나, VS Code의 **Live Server**로 실행합니다.

### ⚙️ Backend 실행

1. `back/` 폴더로 이동하여 IDE(IntelliJ 등)로 프로젝트를 엽니다.
2. 프로젝트 최상단의 `.env.template`을 복사하여 `.env`로 이름을 변경합니다. (기본 H2 설정 사용 시 생략 가능)
3. `BackApplication.java`를 실행하거나 터미널에서 `./gradlew bootRun`을 입력하여 `localhost:8080` 포트로 기동합니다.
4. 기동 시 5개의 제휴사와 6개의 인터페이스 정책이 DB에 자동 세팅됩니다.
