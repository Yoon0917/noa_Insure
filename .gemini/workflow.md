# Workflow & Debugging Rules

## 1. 수정 사항 및 리팩토링 (Modifications & Refactoring)

- 기존 코드를 수정할 때는 "왜 변경하는가?"에 대한 목적을 명확히 하고, 변경 범위(Scope)를 최소화합니다.
- 사이드 이펙트를 방지하기 위해 수정 사항과 관련된 테스트를 먼저 검토하고, 코드 변경 시 관련 부분을 철저히 검증합니다.
- 동작을 변경하지 않는 순수 리팩토링과 기능 추가/버그 수정을 분리하여 진행합니다.

## 2. 트러블슈팅 및 디버깅 (Troubleshooting & Debugging)

- 에러가 발생했을 때 표면적인 증상(Symptom)만 덮지 않고, 근본 원인(Root Cause)을 논리적으로 추적하여 해결합니다.
- 임의의 추측(Guessing)으로 코드를 고치기 전에, 반드시 에러 로그를 확인하고 재현 가능한 시나리오(Reproduction)를 확보합니다.
- 해결 후에는 코드가 더 견고해질 수 있도록 방어적 프로그래밍을 적용합니다.

## [프로젝트 적용 사례]

- **사용자 정의 장애 시뮬레이터**: 단순한 UI 테스트를 넘어, 운영자가 원하는 에러 시나리오(대상 기관, 프로토콜, HTTP 상태 코드, JSON 페이로드 에러 메시지)를 폼을 통해 직접 주입하여 강제로 발생시킬 수 있도록 구현했습니다. 이를 통해 언제든 완벽하게 통제 및 재현 가능한 에러 디버깅 환경을 확보했습니다.
